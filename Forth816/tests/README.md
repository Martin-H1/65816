# Forth816 Regression Tests

This directory contains the regression test framework for the Forth816 project.
Tests are written in Forth using John Hayes's `ttester.fs` framework and run
automatically via a Python script that communicates with the target board over
a serial port.

## Directory Structure

```
tests/
    forth_test.py       Test runner script
    preamble.fs         Loaded before each test file (ttester + DONE + MARKER)
    report.txt          Generated by the test runner (last run)
    masters/            Reference output files (.expected)
    output/             Captured output files (.actual)
    tests/              Test source files (.fs)
        stack_test.fs
        double_test.fs
        ...
```

## Dependencies

- Python 3.11 or later
- [pyserial](https://pypi.org/project/pyserial/) — install with:

```
pip install pyserial
```

## Test File Format

Test files use the `T{` / `->` / `}T` syntax from John Hayes's ttester framework:

```forth
T{ <code> -> <expected stack> }T
```

The framework compares the actual stack contents after `<code>` executes against
the `<expected stack>` values and reports any discrepancy. Each test file must
end with a call to `DONE`:

```forth
T{ 1 2 SWAP -> 2 1 }T
T{ 1 DUP  -> 1 1 }T

DONE
```

### Failure Output

A passing test produces no output. Failures print one of:

```
INCORRECT RESULT: T{ ... -> ... }T
WRONG NUMBER OF RESULTS: T{ ... -> ... }T
```

## Preamble

`preamble.fs` is sent to the board before every test file. It defines:

- The `T{`, `->`, and `}T` test words
- `DONE` — prints the `###DONE###` sentinel and (once `MARKER` is implemented)
  restores the dictionary to its pre-test state

When `MARKER` is available, `preamble.fs` should define:

```forth
MARKER RESET-TEST
: DONE ." ###DONE###" CR RESET-TEST ;
```

This reclaims all dictionary space used by the test file so successive tests
do not exhaust RAM.

## Running the Tests

### Basic usage

From the `tests/` directory:

```
python forth_test.py <port> tests [options]
```

Example on Windows:

```
python forth_test.py COM3 tests --delay 20
```

### Options

| Option | Default | Description |
|---|---|---|
| `--baud BAUD` | 9600 | Serial baud rate |
| `--delay MS` | 10 | Delay between characters in milliseconds |
| `--timeout SEC` | 30 | Seconds to wait for `###DONE###` per test |
| `--preamble FILE` | `preamble.fs` | Preamble filename (relative to test_dir parent) |
| `--no-preamble` | off | Skip preamble for all tests |
| `--master DIR` | `masters/` | Directory containing `.expected` files |
| `--output DIR` | `output/` | Directory for captured `.actual` files |
| `--report FILE` | `report.txt` | Report output path |
| `--generate-masters` | off | Copy captured output to masters as `.expected` files |

### Character delay

The target board's serial I/O can be overrun if characters arrive faster than
the interpreter can process them. The `--delay` option inserts a configurable
pause (in milliseconds) between each transmitted character. Start with `--delay
20` and reduce if tests run too slowly, or increase if you see corrupted output.

## Establishing a Baseline

The first time you run the tests there are no master files to compare against.
Use `--generate-masters` to capture the current output as the reference baseline:

```
python forth_test.py COM3 tests --generate-masters
```

This runs all tests, saves captured output to `output/`, and copies each
`.actual` file to `masters/` as a `.expected` file. Review the `.expected`
files before committing them — they become the reference for all future runs.

To regenerate masters for a single test after an intentional change, copy the
corresponding `.actual` file manually:

```
copy output\double_test.actual masters\double_test.expected
```

## Interpreting Results

The runner prints a one-line status per test as it runs:

```
Running stack_test.fs ...  PASS
Running double_test.fs ...  FAIL
Running math_test.fs   ...  TIMEOUT
```

A summary and full report are written to `report.txt`:

```
Forth816 Regression Test Report
========================================
Tests run:  3
Passed:     2
Failed:     1
Timed out:  0
========================================

double_test: FAIL
    INCORRECT RESULT: T{ 100000. 100000. D+ -> 3464 3 }T
```

For more detail, inspect the captured output in `output/<test_name>.actual`.

### Status meanings

| Status | Meaning |
|---|---|
| `PASS` | No ttester failures and output matches master (if present) |
| `FAIL` | One or more `INCORRECT RESULT` / `WRONG NUMBER OF RESULTS` lines, or output differs from master |
| `TIMEOUT` | `###DONE###` sentinel was not received within the timeout period |
| `MASTER` | Master file written (`--generate-masters` mode only) |

## Adding a New Test File

1. Create `tests/<name>_test.fs` following the format above, ending with `DONE`.
2. Run with `--generate-masters` to capture the initial baseline.
3. Review `masters/<name>_test.expected` to confirm the output is correct.
4. Commit both the test file and the master file.

## Notes

- Test files are run in alphabetical order.
- `preamble.fs` is excluded from the test file list automatically.
- Captured `.actual` files are always written, even for passing tests, so you
  can inspect the full output after any run.
- The `output/` and `masters/` directories are created automatically if they
  do not exist.
